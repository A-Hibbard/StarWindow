# User Leveling System

## Current database findings

Verified against the live Postgres schema on 2026-07-18.

- `users` has account fields plus `status_id`.
- `user_statuses` is the existing level/status table: `status_id`, `status`.
- `user_events` already stores one saved event per user/event with `UNIQUE (user_id, event_id)`.
- `user_events` already has `event_comment` and `event_rating`.
- There is no points, XP, level, achievement, image, or point-history table yet.

This means saving an event is already naturally idempotent. Points should still use a ledger so awards can be audited and so repeat saves/comments cannot grant duplicate points.

Use `user_statuses` as the level definition table:

- `user_statuses.status_id` = level number
- `user_statuses.status` = level title/name
- `users.status_id` = user's current level

Add `user_statuses.min_points` so each level has a point threshold.

## MVP point values

Award points only once per user/event/action unless noted.

| Action | Points | Rule |
| --- | ---: | --- |
| Save event | 10 | First successful save of an event. Resaving later does not award again. |
| Comment on saved event | 20 | First meaningful comment on a saved event. Editing the comment does not award again. |
| Add image to saved event | 30 | First uploaded image for that saved event. Extra images can be added later without MVP points. |
| Rate saved event | 5 | Optional, first rating only. Useful because `event_rating` already exists. |

Recommended MVP complete-event value: `60` points for save + comment + image, or `65` if rating is included.

## Level curve

Level 1 starts at `0` total points.

Use a gentle quadratic curve:

```text
points_needed_for_next_level = 50 * (current_level + 1)
total_points_required_for_level(level) = 25 * level * (level + 1) - 50
```

For level 1, force the total threshold to `0`.

| Level | Total points required | Points from previous level |
| ---: | ---: | ---: |
| 1 | 0 | 0 |
| 2 | 100 | 100 |
| 3 | 250 | 150 |
| 4 | 450 | 200 |
| 5 | 700 | 250 |
| 6 | 1000 | 300 |
| 7 | 1350 | 350 |
| 8 | 1750 | 400 |
| 9 | 2200 | 450 |
| 10 | 2700 | 500 |

Why this curve:

- Level 2 takes about 2 fully engaged saved events, not one tap.
- Level 5 is reachable around 11-12 complete saved events.
- Higher levels slow down without becoming impossible for active users.

## Proposed tables

### user_statuses

Add a threshold column to the existing table.

```sql
ALTER TABLE public.user_statuses
ADD COLUMN IF NOT EXISTS min_points integer NOT NULL DEFAULT 0
CHECK (min_points >= 0);
```

### user_progress

One row per user for fast profile/dashboard reads. The current level stays in `users.status_id`.

```sql
CREATE TABLE public.user_progress (
  user_id integer PRIMARY KEY
    REFERENCES public.users(user_id) ON UPDATE CASCADE ON DELETE CASCADE,
  total_points integer NOT NULL DEFAULT 0 CHECK (total_points >= 0),
  updated_at timestamptz NOT NULL DEFAULT now()
);
```
### point_scores

Config table for point values, so point values are not scattered through code.

```sql
CREATE TABLE public.point_scores (
  action_code varchar(50) PRIMARY KEY,
  display_name varchar(100) NOT NULL,
  points integer NOT NULL CHECK (points >= 0),
  enabled boolean NOT NULL DEFAULT true,
  created_at timestamptz NOT NULL DEFAULT now()
);

INSERT INTO public.point_scores (action_code, display_name, points)
VALUES
  ('save_event', 'Save event', 10),
  ('comment_event', 'Comment on saved event', 20),
  ('add_saved_event_image', 'Add image to saved event', 30),
  ('rate_event', 'Rate saved event', 5);
```

### user_point_event_history

Append-only point ledger. This is the source of truth.

```sql
CREATE TABLE public.user_point_event_history (
  user_point_event_history_id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  user_id integer NOT NULL
    REFERENCES public.users(user_id) ON UPDATE CASCADE ON DELETE CASCADE,
  action_code varchar(50) NOT NULL
    REFERENCES public.point_scores(action_code) ON UPDATE CASCADE ON DELETE RESTRICT,
  points integer NOT NULL CHECK (points >= 0),
  source_type varchar(50) NOT NULL,
  source_key text NOT NULL,
  created_at timestamptz NOT NULL DEFAULT now(),
  UNIQUE (user_id, action_code, source_key)
);

CREATE INDEX idx_user_point_event_history_user_created
  ON public.user_point_event_history (user_id, created_at DESC);
```

Use examples:

- Save event source key: `event:123`
- Comment event source key: `event:123`
- First image source key: `user_event:456:first_image`
- Rating source key: `event:123`

The unique constraint is what prevents repeated saves, comment edits, or image re-uploads from farming points.

### user_event_images

Use a child table instead of adding one image column to `user_events`, because users will naturally expect multiple photos over time.

```sql
CREATE TABLE public.user_event_images (
  user_event_image_id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  user_event_id bigint NOT NULL
    REFERENCES public.user_events(user_event_id) ON UPDATE CASCADE ON DELETE CASCADE,
  image_url text NOT NULL,
  caption text,
  created_at timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX idx_user_event_images_user_event_created
  ON public.user_event_images (user_event_id, created_at DESC);
```

## Comments decision

For MVP, use the existing `user_events.event_comment` column for a private note/comment attached to the saved event.

If the future feature is public event discussion, add a separate table instead:

```sql
CREATE TABLE public.event_comments (
  event_comment_id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  event_id bigint NOT NULL
    REFERENCES public.events(event_id) ON UPDATE CASCADE ON DELETE CASCADE,
  user_id integer NOT NULL
    REFERENCES public.users(user_id) ON UPDATE CASCADE ON DELETE CASCADE,
  body text NOT NULL,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),
  deleted_at timestamptz
);

CREATE INDEX idx_event_comments_event_created
  ON public.event_comments (event_id, created_at DESC);
```

Do not mix public discussion comments with `user_events.event_comment`; that column is better treated as the user's saved-event note.

## Additional mechanics worth adding later

- Mark event as observed: `50` points once per saved event after the event date.
- Add viewing conditions/report: `25` points once per saved event.
- First save of each event type: `15` bonus points to encourage exploration.
- Profile completion: `25` points once for location + event preferences.
- Streak bonus: small weekly bonus for saving or logging observations, capped to avoid farming.

Avoid awarding points for opening screens or refreshing data. Points should reward durable user contributions, not passive app usage.
